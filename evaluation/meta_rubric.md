# Meta Rubric — 元评估量表

---

## 1. 概述

本文档定义了评估体系自身的质量标准——即"尺子的尺子"。用于定期审计 `evaluation/` 目录下所有 Layer 2 rubric 文件的质量，确保评分量表自身满足稳定性、可复现性和完整性的最低要求。

**评估对象**: `evaluation/` 下的任意 Layer 2 rubric 文件（如 `plan_quality_rubric.md`、`memory_quality_rubric.md` 等）

**评估者**: Evaluation Agent (Mode A) 或人工审计

**评估频率**: 每个 rubric 新建/大改后执行一次；每季度定期审计一次

---

## 2. 评分维度与权重

| 维度 | 权重 | 缩写 | 核心问题 |
|------|------|------|---------|
| 锚点校准度 (Anchor Calibration) | 40% | ANC | 评分锚点是否足够精确，能否消除"不同评估者得出不同分数"的问题？ |
| 覆盖完整性 (Coverage Completeness) | 35% | COV | 评分维度是否覆盖了该模块所有关键质量属性？是否遗漏了重要评估面？ |
| 公式自洽性 (Formula Consistency) | 25% | FMC | 计算公式与判定矩阵是否逻辑自洽？计算示例是否与判定结果一致？ |

---

## 3. 锚点校准度 (Anchor Calibration) — 权重 40%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 精确校准 | 每个维度的每级(1-5)均有独立锚点描述 + 最低分/最高分 JSON 示例。两个独立评估者对同一方案打分偏差 ≤ 0.5 分。 |
| 4 | 良好校准 | 每级有文字锚点描述。最低分/最高分有示例。两个评估者偏差 ≤ 1.0 分。 |
| 3 | 基本校准 | 大部分级别有锚点描述，但个别级别模糊（如"一般""较好"等主观词）。缺少 JSON 示例。偏差 ≤ 1.5 分。 |
| 2 | 校准不足 | 多个级别的描述使用模糊词汇。缺少边界案例指导。不同评估者可能给出相差 ≥ 2 分的评价。 |
| 1 | 未校准 | 仅有权重表，无分级锚点描述。完全依赖评估者主观判断。评分结果不可复现。 |

### 检查清单
- [ ] 每个维度是否有 1-5 级独立锚点描述（非跨级复用）？
- [ ] 锚点描述是否使用可观测的判定标准（非主观形容词）？
- [ ] 是否提供最低分（1分）JSON/结构化示例？
- [ ] 是否提供最高分（5分）JSON/结构化示例？
- [ ] 边界案例是否有明确判定规则（如"2.5 分应向上还是向下取整"）？

### 评分指南
```
5分: 5/5 检查要点 ✓
4分: 4/5 检查要点 ✓
3分: 3/5 检查要点 ✓
2分: 2/5 检查要点 ✓
1分: 0-1/5 检查要点 ✓
```

## 4. 覆盖完整性 (Coverage Completeness) — 权重 35%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 全面覆盖 | 评估维度覆盖该模块所有关键质量属性，无已知遗漏。维度权重合理分配（无极端的 <10% 或 >50% 权重）。每个维度有独立检查清单。 |
| 4 | 良好覆盖 | 覆盖核心质量属性 ≥ 90%。权重分配基本合理。大部分维度有检查清单。 |
| 3 | 基本覆盖 | 覆盖核心质量属性 ≥ 70%。存在 1-2 个次要遗漏。权重分配可接受。 |
| 2 | 覆盖不足 | 遗漏 ≥ 1 个核心质量属性或 ≥ 3 个次要属性。权重分配不合理（某维度权重 > 50%）。 |
| 1 | 严重缺失 | 仅覆盖 ≤ 50% 的关键质量属性。缺失整个评估面（如只评功能不评性能）。 |

### 检查清单
- [ ] 维度数量是否在 3-5 个之间（太少=覆盖不足，太多=评估负担过重）？
- [ ] 是否存在"缺失评估面"——该模块有公认的重要属性但 rubric 未覆盖？
- [ ] 各维度权重是否合理（每维度 10%-40%，总和 = 100%）？
- [ ] 每个维度是否有独立的检查清单（≥ 3 个检查要点）？
- [ ] 是否有跨维度重叠（两个维度评同一件事）或遗漏？

### 评分指南
```
5分: 5/5 检查要点 ✓
4分: 4/5 检查要点 ✓
3分: 3/5 检查要点 ✓
2分: 2/5 检查要点 ✓
1分: 0-1/5 检查要点 ✓
```

## 5. 公式自洽性 (Formula Consistency) — 权重 25%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 完全自洽 | 所有计算示例带入公式结果与判定矩阵一致（偏差 ≤ 1 分）。维度地板规则（如有）与判定矩阵无矛盾。公式中所有权重和映射在文档中明确定义。 |
| 4 | 基本自洽 | 核心示例与公式一致。个别边界示例有轻微偏差（≤ 2 分）。公式定义完整。 |
| 3 | 存在偏差 | 至少 1 个示例与公式计算结果偏差 > 2 分但 ≤ 5 分。或公式中某系数/映射未明确定义。 |
| 2 | 明显矛盾 | 多个示例与公式不一致。判定矩阵与公式之间存在逻辑矛盾。 |
| 1 | 严重矛盾 | 公式、示例、判定矩阵三者无法调和。按公式计算的结果与判定矩阵给出的 verdict 完全相反。 |

### 检查清单
- [ ] 加权公式是否明确定义（含各维度权重和缩放因子）？
- [ ] 判定矩阵的分数区间与公式输出范围是否一致？
- [ ] 至少 2 个计算示例是否能手工验算且结果与判定矩阵偏差 ≤ 1 分？
- [ ] 是否存在"公式算出的分数落在判定矩阵未定义的区间"（如公式输出 0-100 但矩阵只覆盖 40-100）？
- [ ] 维度地板/上限规则（如有）是否与判定矩阵逻辑无冲突？

### 评分指南
```
5分: 5/5 检查要点 ✓
4分: 4/5 检查要点 ✓
3分: 3/5 检查要点 ✓
2分: 2/5 检查要点 ✓
1分: 0-1/5 检查要点 ✓
```

## 6. 综合评分计算

### 计算公式
```python
meta_score = (
    ANC_score * 0.40 +
    COV_score * 0.35 +
    FMC_score * 0.25
)
# Range: 1.0 - 5.0 (保留原始 1-5 分制，与 Layer 1 代码质量评分对齐)
```

### 判定矩阵
| 总分范围 | 判定 | 后续行动 |
|---------|------|---------|
| 4.0 - 5.0 | PASS | Rubric 质量达标，可用于生产评估 |
| 3.0 - 3.9 | PASS_WITH_FIXES | 可用但需针对性修复（标注具体维度） |
| 2.0 - 2.9 | NEEDS_REVISION | 必须修订后重新过元评估 |
| 1.0 - 1.9 | REJECT | Rubric 不可用，建议重写 |

## 7. 评估示例

### 示例 1: 高质量 Rubric (plan_quality_rubric.md v1.1.0)

**被评估对象**: `evaluation/plan_quality_rubric.md` (v1.1.0)

**各维度情况**:
- ANC: 5个维度各有1-5级锚点 + 最低分/最高分 JSON 示例。锚点使用可观测判定标准（如"blocking_issues=0"而非"感觉可行"）。→ 5分
- COV: 5个维度(COM/FEA/CON/EXP/ACC)覆盖业务产出全部关键面。权重 25/25/25/15/10 合理。每维度 ≥ 3 个检查要点。→ 5分
- FMC: 含 3 个计算示例，分别手工验算: 例1=87(PASS)✓, 例2=59(REJECT)✓, 例3=52(REJECT)✓。维度地板规则与判定矩阵无矛盾。→ 5分

**计算**:
```
(5×0.40 + 5×0.35 + 5×0.25)
= (2.00 + 1.75 + 1.25)
= 5.00 → PASS
```

### 示例 2: 需修订的 Rubric (假设场景)

**被评估对象**: 某新编写的 `evaluation/xxx_quality_rubric.md` (v0.1.0)

**各维度情况**:
- ANC: 有 1-5 级文字描述但 2 个级别使用"一般""尚可"等模糊词。无 JSON 锚点示例。→ 2分
- COV: 4 个维度，但遗漏了"安全性"这个关键评估面。某维度权重 50% 过高。→ 2分
- FMC: 公式有定义但 1 个示例手工验算偏差 4 分。判定矩阵缺少 20-39 分段。→ 2分

**计算**:
```
(2×0.40 + 2×0.35 + 2×0.25)
= (0.80 + 0.70 + 0.50)
= 2.00 → NEEDS_REVISION
```

**修订建议**:
1. ANC: 为每个维度的 1-5 级补充 JSON 锚点示例；替换模糊词为可观测标准
2. COV: 新增"安全性"维度(权重 15%)，重新分配权重使无极端的 50% 维度
3. FMC: 修正示例计算或公式系数，补全判定矩阵的缺失区间

## 8. 元评估执行流程

### Step 1: 确认评估对象
1. 确定被评估的 rubric 文件路径和版本
2. 确认该 rubric 的目标模块和评估范围

### Step 2: 逐维度评分
1. ANC: 检查每级锚点的精确性，抽样两个评估者对同一案例打分
2. COV: 对照该模块的架构设计文档，确认无遗漏评估面
3. FMC: 手工验算所有计算示例，检查公式、示例、判定矩阵三方一致性

### Step 3: 计算与判定
1. 计算 meta_score
2. 根据判定矩阵给出 verdict
3. 列出具体的修复建议（按优先级排序）

### Step 4: 输出元评估报告
1. 生成 MetaEvaluationReport
2. 记录到评估日志
3. 若 verdict 为 NEEDS_REVISION 或 REJECT，阻塞该 rubric 的生产使用
