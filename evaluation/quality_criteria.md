# Quality Criteria — 总质量准则

---

## 1. 概述

本文档定义了 TravelPlan Orchestrator 项目的**四层质量评估体系**。所有 Agent 的产出都必须经过对应层级的质量评估。

> **当前项目映射**: 当前分支只有 Layer 2（业务产出）的 Plan rubric 正在使用中（用于 ReviewerAgent 评分）。Layer 0/1/3 及 Mode D 为完整质量体系的参考设计，待项目成熟后逐步启用。Layer 2 的其他 7 个 rubric（Memory/Tool/RAG/Reasoning/Protocol/Safety/Evolution）对应的模块当前尚不存在。

```
┌──────────────────────────────────────────────────────────┐
│                    四层质量评估体系                         │
├──────────────────────────────────────────────────────────┤
│ Layer 0: 元评估 (Meta)       → 评估"尺子"自身的质量        │
│ Layer 1: 代码质量 (Code)      → 保证开发质量               │
│ Layer 2: 业务产出 (Business)  → 保证方案质量（8 个模块）    │
│ Layer 3: Agent 贡献度         → 保证系统架构质量           │
│ Mode D:  系统级指标 (System)  → 端到端系统性能             │
└──────────────────────────────────────────────────────────┘
```

---

## 2. Layer 0: 元评估准则 (Meta Evaluation)

### 2.1 评估时机
- 每个 Layer 2 rubric 新建或大改后
- 每季度定期审计

### 2.2 评估维度

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 锚点校准度 | 40% | 评分锚点是否足够精确，能否消除"不同评估者得出不同分数"的问题？ |
| 覆盖完整性 | 35% | 评分维度是否覆盖了该模块所有关键质量属性？ |
| 公式自洽性 | 25% | 计算公式与判定矩阵是否逻辑自洽？计算示例是否与判定结果一致？ |

### 2.3 评分标准
- 5分: 精确校准 + 全面覆盖 + 完全自洽
- 4分: 良好校准 + 核心覆盖 + 基本自洽
- 3分: 基本校准 + 有遗漏 + 存在偏差
- 2分: 校准不足 + 覆盖不足 + 明显矛盾
- 1分: 未校准 + 严重缺失 + 严重矛盾

### 2.4 判定规则
- **总分 ≥ 4.0**: PASS — Rubric 质量达标
- **3.0 ≤ 总分 < 4.0**: PASS_WITH_FIXES — 可用但需针对性修复
- **总分 < 3.0**: NEEDS_REVISION — 必须修订后重新过元评估

**详细量表**: 见 [meta_rubric.md](meta_rubric.md)

## 3. Layer 1: 代码质量准则

### 3.1 评估时机
- 每个 Agent 模块代码首次完成后
- 每次代码变更提交前
- Pull Request 合并前

### 3.2 评估维度

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 正确性 | 30% | 代码逻辑是否正确？是否满足 spec 要求？边界条件是否处理？ |
| 健壮性 | 25% | 异常是否被正确处理？是否有降级策略？输入校验是否完备？ |
| 可读性 | 20% | 命名是否清晰？代码结构是否合理？关键逻辑是否有注释？ |
| 性能 | 15% | 算法复杂度是否合理？是否有明显资源浪费？ |
| 安全性 | 10% | 是否有注入风险？是否校验外部输入？错误信息是否泄露敏感数据？ |

### 3.3 评分标准
- 5分: 优秀 — 超出预期，有额外的质量保障措施
- 4分: 良好 — 满足所有 spec 要求，无明显缺陷
- 3分: 及格 — 基本满足要求，有可优化的点
- 2分: 不足 — 存在明显缺陷，需要修订
- 1分: 严重缺陷 — 存在安全漏洞或逻辑错误

### 3.4 判定规则
- **总分 ≥ 4.0**: PASS — 可以提交
- **3.0 ≤ 总分 < 4.0**: PASS_WITH_SUGGESTIONS — 可以提交但建议后续优化
- **总分 < 3.0**: NEEDS_REVISION — 必须修订后重新评估

## 4. Layer 2: 业务产出准则

### 4.1 评估时机
- 每次 PlannerAgent + KnowledgeAgent 产出完整方案后
- 每次修订迭代后

### 4.2 评估维度

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 完整性 | 25% | 交通、住宿、每日行程、餐饮、预算是否全部覆盖？ |
| 可行性 | 25% | 价格是否合理？时间是否充裕？地理路线是否合理？ |
| 约束满足度 | 25% | 硬约束 100% 满足？软约束 ≥ 70% 满足？ |
| 体验质量 | 15% | 节奏是否合理？活动是否多样？是否个性化？ |
| 信息准确度 | 10% | 价格偏差是否在合理范围？推荐是否基于真实信息？ |

### 4.3 评分标准 (各维度 1-5 分)

**完整性**:
- 5分: 交通 + 住宿 + 每日行程(每天≥2活动) + 餐饮(每天3餐) + 预算明细
- 4分: 缺少 1 个次要元素（如某餐缺推荐）
- 3分: 缺少 1 个主要元素（如没有住宿推荐）
- 2分: 缺少 2 个以上主要元素
- 1分: 仅有框架，缺少实质内容

**可行性**:
- 5分: blocking_issues = 0 且 warnings = 0
- 4分: blocking_issues = 0 且 warnings ≤ 2
- 3分: blocking_issues = 0 且 warnings > 2
- 2分: blocking_issues = 1-2
- 1分: blocking_issues ≥ 3

**约束满足度**:
- 5分: 硬约束 100% + 软约束 ≥ 90%
- 4分: 硬约束 100% + 软约束 80-89%
- 3分: 硬约束 100% + 软约束 70-79%
- 2分: 硬约束 < 100% 或 软约束 < 70%
- 1分: 硬约束多项违反

**体验质量**:
- 5分: 节奏合理 + 活动多样(≥3种类型) + 高度个性化(≥80%偏好匹配)
- 4分: 三项中两项优秀
- 3分: 三项中一项优秀，其他及格
- 2分: 节奏失衡或单调
- 1分: 明显不考虑用户体验

**信息准确度** (基于 KnowledgeAgent 的价格校验):
- 5分: 价格偏差均值 < 10%
- 4分: 价格偏差均值 10-15%
- 3分: 价格偏差均值 15-20%
- 2分: 价格偏差均值 20-30%
- 1分: 价格偏差均值 > 30% 或存在虚构信息

### 4.4 综合得分计算

```
composite_score = (完整性×0.25 + 可行性×0.25 + 约束满足度×0.25 + 体验质量×0.15 + 信息准确度×0.10) × 20
```

范围: 0-100

### 4.5 判定规则
- **≥ 80**: PASS — 质量达标，可以输出
- **60-79**: REVISE — 需要针对性修订
- **< 60**: REJECT — 存在重大缺陷，需重新规划

## 5. Layer 3: Agent 贡献度准则

### 5.1 评估时机
- 每批次任务（≥ 10 个测试用例）完成后
- 每次重大架构变更后
- 定期回归测试（每周）

### 5.2 评估方法

| 方法 | 说明 | 复杂度 |
|------|------|--------|
| LOO 消融实验 | 逐一移除 agent，测量性能变化 Δ | O(n) |
| Agent Importance Score | Peer-rating 聚合 + 排名 | O(n²) |
| 360° 三角评估 | Self-Peer-Supervisory 三方对比 | O(n) |
| 协同效应分析 | 独立 vs 协作性能对比 | O(n) |
| 成本-质量分析 | 单位质量得分的资源成本 | O(n) |

### 5.3 核心指标

**边际贡献 (Marginal Contribution)**:
```
MC_i = S_full - S_no_i
其中 S = 综合质量得分 (0-100)
```

**贡献率 (Contribution Rate)**:
```
CR_i = MC_i / ΣMC_j × 100%
```

**协同增益 (Synergy Gain)**:
```
Synergy = S_full - max(S_planner_alone, S_executor_alone)
正值 = 正协同 (1+1>2), 负值 = 负协同 (互相干扰)
```

**协同效率 (Synergy Efficiency)**:
```
Efficiency = S_full / (S_planner_alone + S_executor_alone) × 100%
> 80% = 强协同, 50-80% = 中等, < 50% = 弱协同
```

**成本-质量比 (Cost-of-Quality)**:
```
CoQ = LLM调用次数 / composite_score
越低越好
```

### 5.4 Agent 标签

| 标签 | 条件 | 含义 |
|------|------|------|
| `veto` | CR > 50% 或移除后系统崩溃 | 不可替代，必须保留 |
| `bottleneck` | CR > 30% 且 Importance Score 最低 | 高负载低评价，需要优化 |
| `standard` | 10% ≤ CR ≤ 50%, 非 bottleneck/veto | 正常贡献 |
| `free_rider` | CR < 10% | 贡献低，可考虑优化或移除 |

### 5.5 告警规则
- Agent 贡献率下降 > 10%（与上次评估对比） → 回归告警
- 协同效率下降 > 20% → Agent 冲突告警
- 360° Self-Peer 偏差持续 > 1.0 → 自我认知偏差告警
- CoQ 上升 > 30% → 成本失控告警

## 6. 质量门 (Quality Gates)

质量准则是质量门的判断依据。质量门的定义和触发规则详见 `gate_definitions.md`。

| Gate | 引用的评估层级 | 通过条件 |
|------|-------------|---------|
| Gate 0 (输入校验) | — | 必填项完整 + 预算 > 0 + 日期合理 |
| Gate 1 (可行性检查) | Layer 2 可行性维度 | blocking_issues = 0 |
| Gate 2 (质量评审) | Layer 2 综合得分 | composite_score ≥ 80 |
| Gate 3 (最终校验) | Layer 2 完整性维度 | 完整性 = 100% + 格式合规 |

## 7. Layer 2 模块级评估索引

以下 8 个模块各有独立的 Layer 2 评估量表（rubric），定义了各自的维度、权重、评分锚点和判定矩阵：

| 模块 | Rubric 文件 | 维度数 | 核心评估面 |
|------|-----------|--------|-----------|
| Plan (方案) | [plan_quality_rubric.md](plan_quality_rubric.md) | 5 | 完整性/可行性/约束满足度/体验质量/信息准确度 |
| Memory (记忆) | [memory_quality_rubric.md](memory_quality_rubric.md) | 4 | 检索命中率/偏好一致性/压缩保真度/跨会话复用率 |
| Tool (工具) | [tool_quality_rubric.md](tool_quality_rubric.md) | 4 | API可用率/响应延迟/结果准确率/降级优雅度 |
| RAG (检索增强) | [rag_quality_rubric.md](rag_quality_rubric.md) | 4 | 检索精度/知识新鲜度/检索延迟/知识覆盖度 |
| Reasoning (推理) | [reasoning_quality_rubric.md](reasoning_quality_rubric.md) | 5 | 约束满足率/幻觉率/自检通过率/修订收敛轮次/推理可追溯性 |
| Protocol (协议) | [protocol_quality_rubric.md](protocol_quality_rubric.md) | 4 | 消息合法率/版本兼容性/错误恢复率/状态转换正确性 |
| Safety (安全) | [safety_quality_rubric.md](safety_quality_rubric.md) | 4 | 有害内容拦截率/误拦截率/注入检测率/PII保护 |
| Evolution (进化) | [evolution_quality_rubric.md](evolution_quality_rubric.md) | 4 | Skill成功率趋势/错误复现率/自动化修复率/知识增长率 |

**Layer 2 评估流程**: 每个模块独立评分 → 加权汇总到 plan_quality_rubric 的综合方案评估中（或独立用于该模块的质量审计）。

**Mode D 系统级指标**: 见 [system_metrics.md](system_metrics.md) — 从 Layer 1/2/3 聚合的端到端系统性能度量。

## 8. 评估一致性保障

### 8.1 评分校准
- 所有评估者使用相同的评分量表
- 定期进行评分一致性检查（同一方案、多人评分、计算 IRR）
- 评分偏差 > 1.0 的评估者需要重新校准

### 8.2 评估日志
- 每次评估结果记录到评估日志
- 日志包含: 评估者、时间、各维度得分、判定、评分依据
- 用于后续趋势分析和基线校准

### 8.3 评估独立性
- 评估 Agent 与被评估 Agent 不能是同一个 Agent
- 评估结果不受被评估方影响
- 评估标准变更需要文档化并通知所有相关方
