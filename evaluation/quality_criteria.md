# Quality Criteria — 总质量准则

---

## 1. 概述

本文档定义了 TravelPlan Orchestrator 项目的**三层质量评估体系**。所有 Agent 的产出都必须经过对应层级的质量评估。

```
┌──────────────────────────────────────────────────┐
│              三层质量评估体系                      │
├──────────────────────────────────────────────────┤
│ Layer 1: 代码质量 → 保证开发质量                   │
│ Layer 2: 业务产出 → 保证方案质量                   │
│ Layer 3: Agent贡献度 → 保证系统架构质量            │
└──────────────────────────────────────────────────┘
```

---

## 2. Layer 1: 代码质量准则

### 2.1 评估时机
- 每个 Agent 模块代码首次完成后
- 每次代码变更提交前
- Pull Request 合并前

### 2.2 评估维度

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 正确性 | 30% | 代码逻辑是否正确？是否满足 spec 要求？边界条件是否处理？ |
| 健壮性 | 25% | 异常是否被正确处理？是否有降级策略？输入校验是否完备？ |
| 可读性 | 20% | 命名是否清晰？代码结构是否合理？关键逻辑是否有注释？ |
| 性能 | 15% | 算法复杂度是否合理？是否有明显资源浪费？ |
| 安全性 | 10% | 是否有注入风险？是否校验外部输入？错误信息是否泄露敏感数据？ |

### 2.3 评分标准
- 5分: 优秀 — 超出预期，有额外的质量保障措施
- 4分: 良好 — 满足所有 spec 要求，无明显缺陷
- 3分: 及格 — 基本满足要求，有可优化的点
- 2分: 不足 — 存在明显缺陷，需要修订
- 1分: 严重缺陷 — 存在安全漏洞或逻辑错误

### 2.4 判定规则
- **总分 ≥ 4.0**: PASS — 可以提交
- **3.0 ≤ 总分 < 4.0**: PASS_WITH_SUGGESTIONS — 可以提交但建议后续优化
- **总分 < 3.0**: NEEDS_REVISION — 必须修订后重新评估

---

## 3. Layer 2: 业务产出准则

### 3.1 评估时机
- 每次 Planning Agent + Execution Agent 产出完整方案后
- 每次修订迭代后

### 3.2 评估维度

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 完整性 | 25% | 交通、住宿、每日行程、餐饮、预算是否全部覆盖？ |
| 可行性 | 25% | 价格是否合理？时间是否充裕？地理路线是否合理？ |
| 约束满足度 | 25% | 硬约束 100% 满足？软约束 ≥ 70% 满足？ |
| 体验质量 | 15% | 节奏是否合理？活动是否多样？是否个性化？ |
| 信息准确度 | 10% | 价格偏差是否在合理范围？推荐是否基于真实信息？ |

### 3.3 评分标准 (各维度 1-5 分)

**完整性**:
- 5分: 交通 + 住宿 + 每日行程(每天≥2活动) + 餐饮(每天3餐) + 预算明细
- 4分: 缺少 1 个次要元素（如某餐缺推荐）
- 3分: 缺少 1 个主要元素（如没有住宿推荐）
- 2分: 缺少 2 个以上主要元素
- 1分: 仅有框架，缺少实质内容

**可行性**:
- 5分: blocking_issues = 0 且 warnings = 0
- 4分: blocking_issues = 0 且 warnings ≤ 2
- 3分: blocking_issues = 0 且 warnings > 2
- 2分: blocking_issues = 1-2
- 1分: blocking_issues ≥ 3

**约束满足度**:
- 5分: 硬约束 100% + 软约束 ≥ 90%
- 4分: 硬约束 100% + 软约束 80-89%
- 3分: 硬约束 100% + 软约束 70-79%
- 2分: 硬约束 < 100% 或 软约束 < 70%
- 1分: 硬约束多项违反

**体验质量**:
- 5分: 节奏合理 + 活动多样(≥3种类型) + 高度个性化(≥80%偏好匹配)
- 4分: 三项中两项优秀
- 3分: 三项中一项优秀，其他及格
- 2分: 节奏失衡或单调
- 1分: 明显不考虑用户体验

**信息准确度** (基于 Execution Agent 的价格校验):
- 5分: 价格偏差均值 < 10%
- 4分: 价格偏差均值 10-15%
- 3分: 价格偏差均值 15-20%
- 2分: 价格偏差均值 20-30%
- 1分: 价格偏差均值 > 30% 或存在虚构信息

### 3.4 综合得分计算

```
composite_score = (完整性×0.25 + 可行性×0.25 + 约束满足度×0.25 + 体验质量×0.15 + 信息准确度×0.10) × 20
```

范围: 0-100

### 3.5 判定规则
- **≥ 80**: PASS — 质量达标，可以输出
- **60-79**: REVISE — 需要针对性修订
- **< 60**: REJECT — 存在重大缺陷，需重新规划

## 4. Layer 3: Agent 贡献度准则

### 4.1 评估时机
- 每批次任务（≥ 10 个测试用例）完成后
- 每次重大架构变更后
- 定期回归测试（每周）

### 4.2 评估方法

| 方法 | 说明 | 复杂度 |
|------|------|--------|
| LOO 消融实验 | 逐一移除 agent，测量性能变化 Δ | O(n) |
| Agent Importance Score | Peer-rating 聚合 + 排名 | O(n²) |
| 360° 三角评估 | Self-Peer-Supervisory 三方对比 | O(n) |
| 协同效应分析 | 独立 vs 协作性能对比 | O(n) |
| 成本-质量分析 | 单位质量得分的资源成本 | O(n) |

### 4.3 核心指标

**边际贡献 (Marginal Contribution)**:
```
MC_i = S_full - S_no_i
其中 S = 综合质量得分 (0-100)
```

**贡献率 (Contribution Rate)**:
```
CR_i = MC_i / ΣMC_j × 100%
```

**协同增益 (Synergy Gain)**:
```
Synergy = S_full - max(S_planner_alone, S_executor_alone)
正值 = 正协同 (1+1>2), 负值 = 负协同 (互相干扰)
```

**协同效率 (Synergy Efficiency)**:
```
Efficiency = S_full / (S_planner_alone + S_executor_alone) × 100%
> 80% = 强协同, 50-80% = 中等, < 50% = 弱协同
```

**成本-质量比 (Cost-of-Quality)**:
```
CoQ = LLM调用次数 / composite_score
越低越好
```

### 4.4 Agent 标签

| 标签 | 条件 | 含义 |
|------|------|------|
| `veto` | CR > 50% 或移除后系统崩溃 | 不可替代，必须保留 |
| `bottleneck` | CR > 30% 且 Importance Score 最低 | 高负载低评价，需要优化 |
| `standard` | 10% ≤ CR ≤ 50%, 非 bottleneck/veto | 正常贡献 |
| `free_rider` | CR < 10% | 贡献低，可考虑优化或移除 |

### 4.5 告警规则
- Agent 贡献率下降 > 10%（与上次评估对比） → 回归告警
- 协同效率下降 > 20% → Agent 冲突告警
- 360° Self-Peer 偏差持续 > 1.0 → 自我认知偏差告警
- CoQ 上升 > 30% → 成本失控告警

## 5. 质量门 (Quality Gates)

质量准则是质量门的判断依据。质量门的定义和触发规则详见 `gate_definitions.md`。

| Gate | 引用的评估层级 | 通过条件 |
|------|-------------|---------|
| Gate 0 (输入校验) | — | 必填项完整 + 预算 > 0 + 日期合理 |
| Gate 1 (可行性检查) | Layer 2 可行性维度 | blocking_issues = 0 |
| Gate 2 (质量评审) | Layer 2 综合得分 | composite_score ≥ 80 |
| Gate 3 (最终校验) | Layer 2 完整性维度 | 完整性 = 100% + 格式合规 |

---

## 6. 评估一致性保障

### 6.1 评分校准
- 所有评估者使用相同的评分量表
- 定期进行评分一致性检查（同一方案、多人评分、计算 IRR）
- 评分偏差 > 1.0 的评估者需要重新校准

### 6.2 评估日志
- 每次评估结果记录到评估日志
- 日志包含: 评估者、时间、各维度得分、判定、评分依据
- 用于后续趋势分析和基线校准

### 6.3 评估独立性
- 评估 Agent 与被评估 Agent 不能是同一个 Agent
- 评估结果不受被评估方影响
- 评估标准变更需要文档化并通知所有相关方
