# RAG Quality Rubric — 知识检索模块质量评分量表

---

## 1. 概述

本文档是 **Layer 2 业务产出评估**中 RAG (Retrieval-Augmented Generation) 模块的专用评分量表。用于评估知识检索系统的检索精度、知识新鲜度、检索延迟和知识覆盖度。

**评估对象**: RAGEngine — 目的地知识库 + 历史方案检索 + 实时价格缓存

**评估时机**: RAG 模块开发完成后、知识库更新后、每季度定期审计

**当前状态**: RAG 模块完全空白。LLM (DeepSeek) 完全依赖自身训练数据生成方案，无外部知识库支撑。此量表用于后续 RAG 模块开发时的评估驱动。

---

## 2. 评分维度与权重

| 维度 | 权重 | 缩写 | 核心问题 |
|------|------|------|---------|
| 检索精度 (Precision) | 30% | PRC | recall@5 / precision@5 / MRR 是否达标？ |
| 知识新鲜度 (Freshness) | 25% | FRS | 价格数据距今天数、景点信息最后更新时间？ |
| 检索延迟 (Speed) | 20% | SPD | 单次检索 P50/P99 延迟是否在目标范围内？ |
| 知识覆盖度 (Coverage) | 25% | CVR | 目的地覆盖率、信息完整度（名称/位置/价格/评价）？ |

---

## 3. 检索精度 (Precision) — 权重 30%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 精确检索 | recall@5 ≥ 0.95, precision@5 ≥ 0.90, MRR ≥ 0.90 |
| 4 | 良好检索 | recall@5 ≥ 0.85, precision@5 ≥ 0.80, MRR ≥ 0.80 |
| 3 | 基本可用 | recall@5 ≥ 0.70, precision@5 ≥ 0.65, MRR ≥ 0.65 |
| 2 | 检索不足 | recall@5 0.50-0.69 或大量无关结果 |
| 1 | 检索失效 | recall@5 < 0.50，返回结果基本不相关 |

### 检查清单
- [ ] recall@5: 前 5 个结果中包含正确答案的比例 ≥ 0.85？
- [ ] precision@5: 前 5 个结果中相关结果占比 ≥ 0.80？
- [ ] MRR: 第一个正确答案的平均排名倒数 ≥ 0.80？
- [ ] 同义查询一致性: "京都寺庙" vs "京都佛寺" 是否返回相似结果？
- [ ] 多语言: 中文查询 vs 英文查询（"Tokyo Tower" vs "东京塔"）的检索质量是否一致？

### 评分指南
```
以 recall@5 为主要依据:
recall@5 ≥ 0.95 → 5分
recall@5 ≥ 0.85 → 4分
recall@5 ≥ 0.70 → 3分
recall@5 0.50-0.69 → 2分
recall@5 < 0.50 → 1分
```

## 4. 知识新鲜度 (Freshness) — 权重 25%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 实时新鲜 | 价格数据 ≤ 3 天, 景点信息 ≤ 30 天, 事件/活动 ≤ 7 天 |
| 4 | 较新鲜 | 价格数据 ≤ 7 天, 景点信息 ≤ 90 天 |
| 3 | 基本新鲜 | 价格数据 ≤ 30 天, 景点信息 ≤ 180 天 |
| 2 | 数据偏旧 | 价格数据 30-90 天, 景点信息 > 180 天 |
| 1 | 严重过时 | 价格数据 > 90 天, 景点信息 > 365 天, 可能包含已关闭/变更的场所 |

### 检查清单
- [ ] 价格数据平均更新间隔 ≤ 7 天？
- [ ] 景点/餐厅的营业状态是否在 90 天内确认过？
- [ ] 是否有"数据过期"标记机制（超过阈值自动标注 stale）？
- [ ] 季节性信息（如赏樱时间、节日活动）是否按年度更新？
- [ ] 是否有自动刷新机制还是依赖手动更新？

### 评分指南
```
综合新鲜度 = (价格新鲜度 × 0.5 + 景点新鲜度 × 0.3 + 事件新鲜度 × 0.2):
综合评分: 全部 ≤ 目标 → 5分
综合评分: 1 项超标 1 级 → 4分
综合评分: 2 项超标 1 级 → 3分
综合评分: 所有项超标 1 级 → 2分
综合评分: 任一项超标 2 级以上 → 1分
```

### 锚点示例

**最低分示例 (1分)** — 数据严重过时:

## 5. 检索延迟 (Speed) — 权重 20%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 极速 | P50 < 50ms, P99 < 200ms |
| 4 | 快速 | P50 < 100ms, P99 < 500ms |
| 3 | 可接受 | P50 < 200ms, P99 < 1s |
| 2 | 偏慢 | P50 200-500ms 或 P99 1-3s |
| 1 | 过慢 | P50 > 500ms 或 P99 > 3s |

### 检查清单
- [ ] 向量检索的索引是否优化（如 HNSW 参数调优）？
- [ ] 是否有结果缓存（相同查询 5 分钟内不重复检索）？
- [ ] embedding 模型推理延迟是否在可接受范围（< 50ms）？
- [ ] 并发检索是否并行执行（非串行等待）？

### 评分指南
```
以 P50 为主要依据:
P50 < 50ms 且 P99 < 200ms → 5分
P50 < 100ms 且 P99 < 500ms → 4分
P50 < 200ms 且 P99 < 1s → 3分
P50 200-500ms 或 P99 1-3s → 2分
P50 > 500ms 或 P99 > 3s → 1分
```

### 锚点示例

**最低分示例 (1分)** — 检索极慢:

**最高分示例 (5分)** — 检索极速:

  }

## 6. 知识覆盖度 (Coverage) — 权重 25%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 全面覆盖 | 目的地覆盖 ≥ 95%（热门 100+ 城市），信息完整度 ≥ 90% |
| 4 | 广泛覆盖 | 目的地覆盖 85-94%，信息完整度 ≥ 80% |
| 3 | 基本覆盖 | 目的地覆盖 70-84%，核心信息完整 |
| 2 | 覆盖不足 | 目的地覆盖 50-69%，信息完整性不足 |
| 1 | 严重不足 | 目的地覆盖 < 50%，仅少数城市有完整信息 |

### 检查清单
- [ ] 热门目的地（全球 Top 100 旅游城市）覆盖率 ≥ 90%？
- [ ] 每个目的地的信息完整度: 是否有名称/位置/描述/价格范围/评分/营业时间？
- [ ] 信息密度: 每个目的地平均有多少个文档/条目？
- [ ] 长尾覆盖: 非热门目的地是否有基础信息（至少有地理坐标）？
- [ ] 信息源多样性: 是否从多个来源交叉验证（非单一来源）？

### 评分指南
```
覆盖率 = 有完整信息的目的地数 / 热门目的地总数 (100):
覆盖率 ≥ 95% 且完整度 ≥ 90% → 5分
覆盖率 85-94% 且完整度 ≥ 80% → 4分
覆盖率 70-84% → 3分
覆盖率 50-69% → 2分
覆盖率 < 50% → 1分
```

## 7. 综合评分计算

### 计算公式
```python
composite_score = (
    PRC_score * 0.30 +
    FRS_score * 0.25 +
    SPD_score * 0.20 +
    CVR_score * 0.25
) * 20
# Range: 20 - 100
```

### 维度地板规则

| 规则ID | 触发条件 | 效果 |
|--------|---------|------|
| FLOOR-PRC | PRC = 1（recall@5 < 50%，检索基本失效） | composite_score = min(composite_score, 59) |
| FLOOR-FRS | FRS = 1（数据严重过时，> 365 天未更新） | composite_score = min(composite_score, 59) |
| FLOOR-MULTI | ≥ 2 个维度得分 < 2 | composite_score = min(composite_score, 69) |

### 判定矩阵
| 总分 | 判定 | 后续行动 |
|------|------|---------|
| 90-100 | EXCELLENT | RAG 模块卓越，检索质量可支撑高质量方案生成 |
| 80-89 | PASS | 质量达标，可投入生产 |
| 70-79 | REVISE (轻) | 针对性优化检索精度或知识覆盖 |
| 60-69 | REVISE (重) | 系统性改进知识库质量或检索性能 |
| 40-59 | REJECT (可修复) | 检索失效或知识严重过时 |
| 20-39 | REJECT (严重) | RAG 模块基本不可用 |

## 8. 评估示例

### 示例 1: 高质量 RAG 系统

**系统概况**: 向量检索 + 实时价格缓存，覆盖 97 个城市
**各维度情况**:
- PRC: recall@5=0.94, precision@5=0.89, MRR=0.88 → 4分
- FRS: 价格 ≤ 2 天, 景点 ≤ 30 天 → 5分
- SPD: P50=35ms, P99=180ms → 5分
- CVR: 覆盖率 97%, 完整度 90% → 5分

**地板规则检查**:
```
PRC=4 (≥2 ✓), FRS=5 (≥2 ✓)
→ 无地板规则触发
```

**计算**:
```
(4×0.30 + 5×0.25 + 5×0.20 + 5×0.25) × 20
= (1.20 + 1.25 + 1.00 + 1.25) × 20
= 4.70 × 20
= 94 → EXCELLENT
```

### 示例 2: 低质量 RAG 系统

**系统概况**: 无向量索引，全量扫描，知识库过时
**各维度情况**:
- PRC: recall@5=0.35, 关键词匹配 → 1分
- FRS: 价格数据 150 天, 景点信息 2 年 → 1分
- SPD: P50=800ms, 全量扫描 → 1分
- CVR: 覆盖率 30%，仅 15 个城市 → 1分

**地板规则检查**:
```
PRC=1 → FLOOR-PRC 触发！
FRS=1 → FLOOR-FRS 触发！
维度 <2 = 4 → FLOOR-MULTI 触发！
→ composite_score 上限 = 59
```

**计算**:
```
(1×0.30 + 1×0.25 + 1×0.20 + 1×0.25) × 20
= (0.30 + 0.25 + 0.20 + 0.25) × 20
= 1.00 × 20
= 20 → min(20, 59) = 20
→ REJECT (严重)
```
