# Code Quality Rubric — 代码质量评分量表

---

## 1. 概述

本文档是 **Layer 1 代码质量评估**的详细评分量表（Rubric）。Evaluation Agent 在 Mode A 下使用此量表对 Agent 代码进行评分。

每个维度包含 5 个等级的具体描述和评分锚点，确保评估的一致性和可重复性。

---

## 2. 评分维度与权重

| 维度 | 权重 | 缩写 |
|------|------|------|
| 正确性 (Correctness) | 30% | COR |
| 健壮性 (Robustness) | 25% | ROB |
| 可读性 (Readability) | 20% | REA |
| 性能 (Performance) | 15% | PER |
| 安全性 (Security) | 10% | SEC |

---

## 3. 正确性 (Correctness) — 权重 30%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 卓越 | 逻辑完全正确，覆盖所有边界条件、异常路径和 spec 要求。有完整的单元测试覆盖。 |
| 4 | 良好 | 核心逻辑正确，满足 spec 要求。边界条件已处理但可能存在个别遗漏。有基本的单元测试。 |
| 3 | 及格 | 主要逻辑正确，happy path 通过。部分边界条件未处理。测试覆盖不足。 |
| 2 | 不足 | 存在明显的逻辑错误或 spec 偏离。部分功能在特定输入下会产生错误输出。 |
| 1 | 严重 | 核心逻辑有严重缺陷。多数测试用例失败。无法满足基本功能需求。 |

### 检查要点
- [ ] 是否满足 spec 中定义的所有功能需求？
- [ ] 边界条件是否正确处理（空输入、极值、null）？
- [ ] 状态转换是否正确（状态机逻辑）？
- [ ] 是否有单元测试？覆盖率 ≥ 70%？
- [ ] 是否与 spec 中的输入/输出 Schema 一致？

### 评分指南
```
5分: 所有检查要点 ✓，且有超出 spec 的额外保障
4分: 所有核心检查要点 ✓，1-2 个次要项有遗漏
3分: 核心检查要点 ✓，3-4 个次要项有遗漏
2分: 1 个核心检查要点 ✗
1分: 2+ 个核心检查要点 ✗
```

---

## 4. 健壮性 (Robustness) — 权重 25%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 卓越 | 完整的异常处理体系。所有外部调用有超时+重试。有降级策略。输入校验完备。错误信息清晰可操作。 |
| 4 | 良好 | 主要异常路径已处理。外部调用有超时设置。基本输入校验。错误信息清晰。 |
| 3 | 及格 | 有基本的 try-catch/异常处理。但覆盖不全。部分外部调用无超时。错误信息基本可用。 |
| 2 | 不足 | 异常处理零散。缺乏系统性的错误处理策略。部分崩溃场景未防护。 |
| 1 | 严重 | 几乎没有异常处理。输入不校验。单点故障可导致整个系统崩溃。 |

### 检查要点
- [ ] 所有外部 API 调用是否有超时设置（≤ 30s）？
- [ ] 是否有重试机制（含指数退避）？
- [ ] 输入参数是否校验（类型、范围、非空）？
- [ ] 是否有降级策略（关键服务不可用时的 Plan B）？
- [ ] 错误信息是否包含足够上下文？
- [ ] 是否避免了吞掉异常（bare except）？

### 评分指南
```
5分: 6/6 检查要点 ✓
4分: 5/6 检查要点 ✓
3分: 4/6 检查要点 ✓
2分: 2-3/6 检查要点 ✓
1分: 0-1/6 检查要点 ✓
```

---

## 5. 可读性 (Readability) — 权重 20%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 卓越 | 命名精准、代码结构清晰、关键逻辑有恰到好处的注释。符合 PEP 8 等语言规范。函数职责单一。 |
| 4 | 良好 | 命名清晰、结构合理。大部分关键逻辑有注释。基本符合语言规范。 |
| 3 | 及格 | 命名基本可理解。结构可接受但有改进空间。注释偏少。 |
| 2 | 不足 | 命名模糊（如 x, tmp, data）、结构混乱、缺乏注释。 |
| 1 | 严重 | 单字母变量、无注释、代码组织混乱。难以维护。 |

### 检查要点
- [ ] 变量/函数/类命名是否表意清晰？
- [ ] 函数长度是否合理（≤ 50 行）？
- [ ] 类职责是否单一（SRP）？
- [ ] 关键算法/复杂逻辑是否有注释？
- [ ] 魔术数字是否提取为常量？
- [ ] 是否符合 PEP 8 (Python) 或对应语言规范？

### 评分指南
```
5分: 6/6 检查要点 ✓
4分: 5/6 检查要点 ✓
3分: 3-4/6 检查要点 ✓
2分: 2/6 检查要点 ✓
1分: 0-1/6 检查要点 ✓
```

---

## 6. 性能 (Performance) — 权重 15%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 卓越 | 算法经过优化，使用合适的数据结构。有缓存策略。异步/并行处理得当。有性能基准测试。 |
| 4 | 良好 | 算法复杂度合理。无明显性能瓶颈。基本的数据结构选择合理。 |
| 3 | 及格 | 功能正确但存在可优化的性能点。缺少缓存。 |
| 2 | 不足 | 存在明显性能问题（如不必要的 O(n²) 操作、重复计算）。 |
| 1 | 严重 | 严重的性能问题导致不可用。如超大列表的嵌套循环、频繁的同步 I/O。 |

### 检查要点
- [ ] 关键路径算法复杂度是否合理？
- [ ] 是否有不必要的重复计算或 I/O？
- [ ] 是否有缓存策略（对重复查询）？
- [ ] 是否合理使用异步/并行处理？
- [ ] 大数据量场景是否考虑？

### 评分指南
```
5分: 5/5 检查要点 ✓ + 有性能测试
4分: 4/5 检查要点 ✓
3分: 3/5 检查要点 ✓
2分: 2/5 检查要点 ✓
1分: 0-1/5 检查要点 ✓
```

## 7. 安全性 (Security) — 权重 10%

### 评分锚点

| 分数 | 描述 | 判定标准 |
|------|------|---------|
| 5 | 卓越 | 全面的安全防护。输入/输出编码。敏感数据保护。依赖安全检查。无已知漏洞。 |
| 4 | 良好 | 基本安全措施到位。输入校验。无明显的注入风险。 |
| 3 | 及格 | 主要安全风险已处理。但存在个别风险点。 |
| 2 | 不足 | 存在可识别的安全漏洞。缺少输入过滤。 |
| 1 | 严重 | 明显安全漏洞。如 SQL/命令注入、明文密码、未校验的外部输入。 |

### 检查要点
- [ ] 外部输入是否经过校验和过滤？
- [ ] 是否存在注入风险（命令注入、路径遍历等）？
- [ ] 敏感信息（密钥、密码）是否硬编码？
- [ ] 错误信息是否泄露内部实现细节？
- [ ] 依赖库是否有已知漏洞（CVE）？

### 评分指南
```
5分: 5/5 检查要点 ✓
4分: 4/5 检查要点 ✓
3分: 3/5 检查要点 ✓
2分: 2/5 检查要点 ✓
1分: 0-1/5 检查要点 ✓，或存在高危漏洞
```

## 8. 综合评分计算

### 计算公式
```python
total_score = (
    COR_score * 0.30 +
    ROB_score * 0.25 +
    REA_score * 0.20 +
    PER_score * 0.15 +
    SEC_score * 0.10
)
```

### 判定矩阵
| 总分范围 | 判定 | 后续行动 |
|---------|------|---------|
| 4.0 - 5.0 | PASS | 可以提交/合并 |
| 3.0 - 3.9 | PASS_WITH_SUGGESTIONS | 可以提交，创建后续优化 issue |
| 2.0 - 2.9 | NEEDS_REVISION | 必须修订，重新评估 |
| 1.0 - 1.9 | REJECT | 重大缺陷，建议重写 |

## 9. 评估流程

### Step 1: 准备
1. 加载目标代码文件
2. 确认关联的 spec 文件（对照检查正确性）
3. 准备评分表

### Step 2: 逐维度评分
1. 按正确性 → 健壮性 → 可读性 → 性能 → 安全性顺序
2. 每个维度独立评分，不受其他维度影响
3. 记录每个维度的具体 issue 和 evidence

### Step 3: 计算与判定
1. 计算加权总分
2. 根据判定矩阵给出 verdict
3. 按优先级整理 action_items

### Step 4: 输出报告
1. 生成 CodeQualityReport JSON
2. 记录到评估日志
3. 发送给 Orchestrator

## 10. 常见评分误区

| 误区 | 正确做法 |
|------|---------|
| 因代码"看起来很漂亮"给高分 | 按检查要点逐项验证，不可仅凭直觉 |
| 因一个严重缺陷全线给低分 | 各维度独立评分，一个维度的缺陷不影响其他维度 |
| 因作者是资深开发者给高分 | 评分只看代码本身，不看来历 |
| 评分趋中（全给3分） | 使用评分锚点，有依据地给出差异化分数 |
| 只看代码不看 spec | 代码的正确性必须对照 spec 验证 |
